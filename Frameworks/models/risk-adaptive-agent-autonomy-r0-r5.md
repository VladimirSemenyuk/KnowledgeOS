# Risk-adaptive agent autonomy R0-R5

Risk-adaptive agent autonomy R0-R5 - лестница разрешений, которая задает допустимую автономию агента в зависимости от контекста, риска и обратимости действий.

## Коротко

Уровень автономии агента не должен быть статичным. Один и тот же агент может иметь:

- R1 в production;
- R2 в feature branch;
- R3 при полном eval coverage и шаблоне из Pattern Library;
- R5 в изолированной песочнице.

Автономность должна пересчитываться в runtime, а не назначаться один раз в политике.

## Лестница R0-R5

| Уровень | Что разрешено | Когда применим |
| --- | --- | --- |
| R0 | только чтение | production без проверенных evals |
| R1 | только предложения, решение принимает человек | критичные пути production |
| R2 | действия в feature branch с PR-review | стандартная разработка |
| R3 | auto-merge при прохождении evals | стандартизированные шаблоны, надежная Pattern Library |
| R4 | выполнение в нескольких сессиях с checkpoints | зрелые long-running сценарии |
| R5 | полная автономия в песочнице | изолированные эксперименты |

## Триада адаптации

Фактический уровень автономии определяется триадой:

| Фактор | Низкое значение | Высокое значение |
| --- | --- | --- |
| Критичность для бизнеса | внутренние инструменты, support apps | клиентские приложения, конкурентное ядро |
| Профиль риска | нет ПДН, транзакций, регулируемых данных | ПДН, финансовые операции, security-critical paths |
| Архитектурная сложность | один модуль, локальное изменение | несколько интеграций, распределенное состояние |

Формула из концепции:

```text
effective_R = nominal_R - criticality_raise - risk_raise - complexity_raise
```

Если задача номинально R3, но имеет высокую бизнес-критичность и высокий профиль риска, фактический уровень может снизиться до R0-R1.

## Человек в цикле и человек над циклом

| Режим | Где применим | Смысл |
| --- | --- | --- |
| Человек в цикле | R0-R2 | человек участвует в существенных точках решения |
| Человек над циклом | R3-R5 | человек наблюдает, вмешивается при исключениях |

R3 - граница переключения режима. Поэтому R3 требует особенно сильных evals, policy hooks и объяснимости.

## Как это исполняется

На каждой агентной операции policy hook bus должен оценивать:

- среду: production / staging / dev / sandbox;
- ветку: main / release / feature / experiment;
- критичность файла или модуля;
- обратимость действия;
- покрытие тестами;
- радиус воздействия;
- статус [[Frameworks/models/evidence-bundle|Evidence Bundle]];
- длину task horizon.

Результат определяет: разрешить, запросить подтверждение, заблокировать, откатить или эскалировать.

## Антипаттерн

Статичная лестница R0-R5 без runtime-пересчета создает один из двух провалов:

- чрезмерный контроль: все идет через ручные approvals, скорость теряется;
- недостаточный контроль: клиентские или регулируемые сценарии получают автономию, допустимую только для песочницы.
